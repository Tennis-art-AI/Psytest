## 0. КРАСНЫЕ ЛИНИИ

MentalScreenLab — информационный скрининговый сервис. Мы не ставим диагнозы, не назначаем лечение, не оказываем медицинскую помощь.

### Что ЗАПРЕЩЕНО в любой диагностике

**Препараты и лечение:**
- Названия препаратов — ни МНН, ни бренды (литий, ламотриджин, сертралин, кветиапин и любые другие)
- Группы препаратов как рекомендация (СИОЗС, антипсихотики, стабилизаторы настроения, бензодиазепины, антидепрессанты)
- Дозировки — никакие, ни в каком контексте
- Названия методов терапии в выводах (КПТ, ЭПР, IPSRT, ДПТ, ДПДГ) — запрещены в секции результатов. Допустимы ТОЛЬКО в психообразовательных блоках (qnt) внутри вопросов, где это контекст вопроса, а не рекомендация
- Схемы лечения: «стабилизатор + антипсихотик», «СИОЗС + ЭПР»

**Формулировки:**
- Директивные: «рекомендуется», «первая линия», «препарат выбора», «показан», «эффективен при»
- Диагностические: «это [расстройство]», «у вас [расстройство]», «паттерн совпадает с [расстройством]»

### Что ОБЯЗАТЕЛЬНО в каждой диагностике

1. **Экран согласия** (перед первым вопросом): это скрининг — не диагноз; не заменяет специалиста; данные не покидают браузер; возрастное предупреждение (<18 — с родителем)
2. **Кризисный блок** — статичный, на каждой странице: Телефон доверия 8-800-2000-122
3. **Блок «К специалисту»** — при уровне ≥ лёгкого: психиатр, психотерапевт, клинический психолог
4. **Дисклеймер в выводах:** «Результат скрининга — не диагноз. Только специалист может поставить диагноз на основании очного осмотра»
5. **Дисклеймер-футер** на каждой странице

### Допустимые формулировки

| Запрещено | Допустимо |
|---|---|
| Это [расстройство] | Ваши ответы указывают на признаки, которые специалист может расценить как [расстройство] |
| Рекомендуется [препарат/метод] | Существуют эффективные методы помощи. Обратитесь к специалисту |
| Поддаётся лечению | Поддаётся лечению при обращении к специалисту |
| Первая линия / препарат выбора | *(убрать, не заменять)* |
| Диагностика | Скрининг |
| Диагноз | Результат скрининга |
| Лечение, назначение | Помощь, поддержка |

### Допустимый контент в выводах

- Описание: расстройство поддаётся лечению **при обращении к специалисту** (всегда с оговоркой)
- К кому обращаться: психиатр, психотерапевт, клинический психолог
- Кросс-диагностические ссылки (пройдите также скрининг X)
- Научные факты: эпидемиология, наследуемость, течение — это психообразование
- Механизмы расстройства (модели) — психообразование
- Слово «антидепрессант» в вопросе истории (принимали ли?) — допустимо (вопрос о прошлом опыте, не рекомендация)

### Чеклист перед деплоем (grep по файлу)

- [ ] Нет названий препаратов
- [ ] Нет групп препаратов
- [ ] Нет дозировок (мг, mg)
- [ ] Нет директивных формулировок
- [ ] Нет методов терапии в выводах (только в qnt-блоках)
- [ ] Есть экран согласия
- [ ] Есть кризисный блок (статичный)
- [ ] Есть дисклеймер в выводах
- [ ] При уровне ≥ лёгкого есть «обратитесь к специалисту»
- [ ] МКБ-11 и DSM-5-TR коды указаны

## 1. ОСНОВА ДИАГНОСТИКИ: МЕЖДУНАРОДНЫЕ ИНСТРУМЕНТЫ

Каждая диагностика строится НЕ из головы, а на базе существующих международных валидированных инструментов, стандартов и клинических моделей. Это фундамент — без него диагностика невалидна.

### Что обязательно изучить перед созданием диагностики

1. **DSM-5-TR критерии (APA 2022)** + **МКБ-11 критерии (ВОЗ 2022)** — полные, со спецификаторами и исключениями. Россия использует МКБ — её коды обязательны
2. **3–6 валидированных инструментов** конкретного расстройства — с разбором по пунктам, подшкалам, cut-off значениям, чувствительности/специфичности, психометрическими свойствами
3. **Клинические модели** — когнитивные, поведенческие, нейробиологические — почему расстройство поддерживается
4. **Клинические гайдлайны** (NICE, APA, WFSBP, BAP) — для понимания лечебных импликаций (НЕ для включения в выводы, но для понимания, что важно выявить)
5. **Дифференциальная диагностика** — от каких расстройств отличать, по каким признакам
6. **Феноменология** — как пациенты описывают свой опыт (для формулировок вопросов)
7. **Ловушки самодиагностики** — что именно при этом расстройстве мешает точному самоотчёту

### Примеры инструментов по расстройствам

| Расстройство | Ключевые инструменты |
|---|---|
| ГТР | GAD-7, PSWQ, IUS-12, MCQ-30, HAM-A, DSM-5-TR |
| Депрессия | PHQ-9, BDI-II, HAM-D17, MADRS, IDS-30, DSM-5-TR |
| БАР | MDQ, HCL-32, YMRS, PHQ-9 (депрессивный полюс), Altman Self-Rating Mania, DSM-5-TR |
| ОКР | Y-BOCS/Y-BOCS-II, OCI-R, DOCS, FOCI, OCI-4, BABS (insight), DSM-5-TR, МКБ-11 |

**Каждый вопрос диагностики ОБЯЗАН иметь источник** — конкретный пункт конкретного инструмента или критерий DSM-5/МКБ-11. Источник указывается как тег вопроса.

### Откуда брать клинические данные

Для каждого расстройства существует (или создаётся) **база знаний** — файл `docs/section_[N]_[disorder].md`. Это единственный источник клинической правды. Если база знаний есть — все клинические решения (домены, вопросы, пороги, history, domain-ratio, кросс-ссылки) берутся из неё.

**Готовые базы знаний:**
- `docs/section1_GAD.md` — ГТР
- `docs/section2_depression.md` — Депрессия (БДР)
- `docs/section3_BAR.md` — БАР
- `docs/section4_OCD.md` — ОКР

Если базы знаний нет — она создаётся ПЕРЕД созданием диагностики через протокол Фаз-0 (§5).

## 2. ОТСУТСТВИЕ МЕДИЦИНСКИХ РЕКОМЕНДАЦИЙ В ВЫВОДАХ

Отдельно и явно: выводы диагностики (функция buildC() в config-файле) НЕ содержат:

- Названий препаратов или групп препаратов
- Названий методов терапии (КПТ, ЭПР, ДПДГ, ДПТ и т.д.)
- Рекомендаций «вам нужно [конкретное лечение]»
- Противопоказаний («антидепрессанты опасны при...»)

Выводы СОДЕРЖАТ:
- Уровень тяжести (5 уровней)
- Доменные комментарии (какие области затронуты)
- Перенаправление к специалисту (при уровне ≥ лёгкого)
- Кросс-диагностические ссылки (пройдите также скрининг X)
- Обработку истории (длительность, коморбидность, семейный анамнез)
- Domain-ratio флаги (если паттерн ответов указывает на альтернативный диагноз)
- Психообразование (факты о расстройстве, течении, распространённости)

**Контекст:** вся информация о лечении нужна при Фаз-0 исследовании — для ПОНИМАНИЯ, что важно выявить. Но в сам скрининг эта информация не попадает. Мы готовим человека к визиту к специалисту, а не заменяем специалиста.

## 3. КРОСС-ССЫЛКИ И ДИФФЕРЕНЦИАЛЬНАЯ ДИАГНОСТИКА

### Принцип

Расстройства часто похожи друг на друга. Скрининг должен не только оценить тяжесть, но и указать, если паттерн ответов больше похож на другое расстройство.

### Механизмы

**Domain-ratio:** отношение балла домена к его максимуму (0.0–1.0). Если один домен аномально высокий при другом аномально низком — флаг альтернативного диагноза. Формула: `ratioX = dS(X) / DM.X`. Флаг: `ratioX > 0.6 AND ratioY < 0.3`.

**История:** вопросы X2, X3 (дифдиагностика) напрямую спрашивают о признаках конкурирующих расстройств.

**Кросс-ссылки:** при выявлении флага или релевантного ответа истории — блок «Пройдите также скрининг [X]» со ссылкой.

### Где брать конкретные паттерны

Из базы знаний расстройства:
- `BRIEF: Обоснование domain-ratio флагов` — какие паттерны на что указывают
- `Дифференциальная диагностика` — таблицы сравнения с конкурирующими расстройствами
- `BRIEF: Обработка истории` — какой вариант ответа на что указывает

Если расстройство A может быть спутано с расстройством B, и у обоих есть базы знаний — смотреть ОБЕ базы для полной картины дифференциации.

## 4. ТЕХНИЧЕСКАЯ АРХИТЕКТУРА ДИАГНОСТИКИ

**Это рекомендация, не обязаловка.** Каждая диагностика прорабатывается отдельно. Но текущая архитектура работает и единообразна — менять без причины не стоит.

### Файловая структура

Каждая диагностика = 1 HTML-файл (страница) + 1 JS config-файл:
- `screening/[disorder]/index.html` — страница диагностики
- `config-[disorder].js` — вся клиническая логика (вопросы, домены, пороги, buildC, history)
- `engine.js` — общий движок (обработчики, подсчёт баллов, UI-обновление)
- `common.css` — общие стили

### Структура config-файла

```
1. Константы: TQ, FQ, MAX, SCORED, DOM, DM, LEV, DESC
2. buildC() — функция выводов (домены, history, domain-ratio, кросс-ссылки)
3. Вопросы — в HTML (data-атрибуты), не в JS
```

**Что определяют константы:**
- `TQ` — scored вопросов, `FQ` — функционирование (вне балла)
- `MAX` — максимальный балл scored-доменов
- `DOM` — маппинг доменов к номерам вопросов: `{ A:[1,5], B:[6,10], C:[11,14], D:[15,18], E:[19,21] }`
- `DM` — максимум каждого домена: `{ A:20, B:20, C:16, D:16, E:12 }`
- `LEV` — 5 уровней тяжести с порогами, цветами
- `DESC` — текстовые описания 5 уровней

### Текущий стандарт (может меняться)

- **5 доменов (A–E)**, E = функционирование (вне балла)
- **5 уровней тяжести**, пороги равномерные: `step ≈ MAX / 5`
- **Anti-bias:** значение 0 не на 1-й позиции, максимум не на последней, «плавающий» не на 3-й. Нумерация — числа (1,2,3...), не буквы
- **Психообразовательные блоки (qnt):** 2–3 на диагностику, для непонятных концепций. Методы терапии допустимы ТОЛЬКО здесь
- **Высокорисковые ответы (sel-w):** визуальное выделение при выборе опасного ответа (суицидальность, психоз)
- **История:** 6 слотов (X1 течение, X2/X3 дифдиагностика, Д1 ремиссии, Д2 коморбидность, Д3 семейный анамнез) + дополнительные слоты для DSM-5 спецификаторов при необходимости

### HTML-блоки страницы (порядок)

1. Consent-экран (согласие)
2. Кризисный блок (статичный)
3. Intro: «Нормальное vs Расстройство» (2 карточки)
4. Инструкция оценки
5. Вопросы (scored + функционирование)
6. История (дифдиагностика)
7. Результаты (баллы, домены, выводы)
8. Дисклеймер-футер

### Технический референс

`config-bar.js` — наиболее полная реализация (с sub-module). Использовать ТОЛЬКО как пример HTML/JS/CSS паттернов. **Категорически запрещено** копировать клиническое содержание (вопросы, домены, пороги, выводы) из одной диагностики в другую. Клиника берётся из базы знаний.

## 5. ПОСЛЕДОВАТЕЛЬНОСТЬ СОЗДАНИЯ НОВОЙ ДИАГНОСТИКИ

### Фаз-0: Клиническое исследование

**Обязательно. Не начинать код без завершения.**

Провести web search по каждому пункту, собрать результаты в базу знаний (`docs/section_[N]_[disorder].md`):

1. **DSM-5-TR + МКБ-11:** критерии, спецификаторы, исключения, временные критерии
2. **Валидированные инструменты (3–6):** пункты, подшкалы, cut-off, чувствительность/специфичность, что покрывает, что не покрывает
3. **Клинические модели:** когнитивные, поведенческие модели — почему расстройство поддерживается
4. **Феноменология:** как пациенты описывают опыт, метафоры, что скрывают, заблуждения
5. **Лечение (гайдлайны):** NICE, APA, WFSBP, BAP — для понимания, что важно выявить (НЕ для включения в выводы)
6. **Дифференциальная диагностика:** для каждой конкурирующей пары — ключевой признак, на каком вопросе проявится, при каком паттерне
7. **Ловушки самодиагностики:** что конкретно при этом расстройстве мешает точному самоотчёту
8. **Коморбидность:** топ-3 коморбидных диагноза с процентами и источниками
9. **Эпидемиология, генетика, нейробиология, гендерные и возрастные различия**

**Результат:** база знаний — показать пользователю, получить одобрение.

### Фаз-1: Архитектура

На основе базы знаний определить:
- Домены (A–E): что измеряет каждый, шкала, количество вопросов
- Пороги тяжести (5 уровней)
- Sub-module (если расстройство имеет подтипы)
- Слоты истории (X1–Д3 + дополнительные для спецификаторов)
- Domain-ratio флаги
- Кросс-ссылки
- Дифференциальную таблицу (5 параметров × 4 расстройства)
- Intro-карточки (нормальное vs расстройство)

**Показать пользователю → одобрение.**

### Фаз-2: Реализация

Код по технической архитектуре (§4). Каждый вопрос = обоснование из базы знаний. MCP: write → verify → backup.

### Фаз-3: Верификация

- Каждый вопрос → источник (инструмент + критерий из базы знаний)
- Каждый вывод → допустимость по §0
- Domain-ratio → клинический смысл из базы знаний
- Ловушки самодиагностики → учтены в формулировках
- Чеклист §0 пройден (grep)
- Финальная проверка + backup
